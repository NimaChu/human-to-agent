# PR：Agent 工作区渐进演化机制

> **文档类型**：产品需求 / 方法论 PR  
> **目标读者**：Codex、平台开发者、业务负责人、使用 OpenCode/Codex 的一线员工  
> **核心目标**：建设一个“母工作区”，引导员工将一个真实的人类工作，经过五个阶段持续补充 unknown、显化知识、固化能力，最终生成一个具备进入 Loop 条件的完备子工作区（Loop-ready Harness）。

---

## 1. 背景

企业已经开始使用 OpenCode/Codex 完成各类工作，但目前大多数使用仍停留在：

- 临时对话；
- 单次任务；
- 个人经验；
- 局部脚本；
- 不可复用的提示词；
- 依赖操作者补充大量隐性上下文；
- 成功一次，但无法稳定交给其他人或系统重复执行。

真正有价值的方向，不是让员工“多使用几次 AI”，而是将员工实际完成的工作逐步沉淀为组织可复用、可验证、可组合、可运营的 Agent 工作资产。

本 PR 提议建设一个用于“生产子工作区的工作区”，即：

> **Agent 工作区孵化器 / Harness Builder Workspace**

员工不需要一开始就完整描述业务，也不需要一次性设计出一个 Agent。系统通过五个阶段，引导员工选择一个小任务、复现任务、发现 unknown、补充证据、固化 Skill、组合 Harness，并在满足条件后交付一个具备 Loop 基础的子工作区。

该方法的核心不是“一次设计正确”，而是：

> **允许从不完整开始，通过真实使用不断发现 unknown，并把 unknown 逐步转化为明确的知识、规则、案例、边界和机制。**

---

## 2. 要解决的问题

### 2.1 业务工作大量依赖隐性知识

员工能够完成任务，并不代表员工能够一次性说明：

- 为什么这样做；
- 哪些输入是必要的；
- 哪些规则是默认规则；
- 哪些情况需要人工判断；
- 什么结果才算正确；
- 哪些异常最危险；
- 哪些操作不可逆；
- 哪些信息只存在于个人经验中。

因此，直接要求员工“编写一个完备 Skill”通常不现实。

### 2.2 单次成功不能证明可复用

一个任务在某次对话中成功，可能依赖：

- 特定文件；
- 特定上下文；
- 员工临时纠错；
- 某个模型的偶然输出；
- 未记录的人工判断；
- 只覆盖了正常案例。

需要将一次成功逐步转化为：

- 可重复；
- 可测试；
- 可解释；
- 可移交；
- 可组合；
- 可恢复。

### 2.3 Skill 容易成为“提示词收藏”

缺少统一方法时，Skill 可能出现：

- 边界不清；
- 粒度不一致；
- 无测试案例；
- 不知道何时适用；
- 不知道何时必须停止；
- 不知道依赖什么输入；
- 不知道由谁维护；
- 只能由创建者本人使用。

### 2.4 Harness 容易退化为 Skill 的简单堆叠

多个 Skill 放在一起，并不会自然形成一个可靠的工作系统。

一个可运行的 Harness 还需要：

- 明确目标；
- 公共上下文；
- 工作步骤；
- 状态；
- 判断节点；
- 权限；
- 验收机制；
- 异常处理；
- 人工接管；
- 运行记录。

### 2.5 Loop 容易被误解为“重复运行”

真正的 Loop 不是不断重新执行，而是一个有界闭环：

- 知道目标；
- 知道当前状态；
- 知道距离目标还有什么差距；
- 能选择下一步行动；
- 能验证行动结果；
- 能停止；
- 能升级给人；
- 能控制风险和成本。

因此，在进入 Loop 前，必须先形成一个 **Loop-ready Harness**。

---

## 3. 产品愿景

建设一个统一的母工作区，使任何业务员工都能从一个真实小任务出发，通过系统引导完成以下演化：

```text
真实工作
→ 工作复现
→ 任务契约
→ Skill
→ 多案例验证
→ Harness
→ Loop-ready Harness
```

最终，不同同事可以用同一个母工作区，为自己的工作创建不同的子工作区。

例如：

```text
母工作区
├── 产能分析子工作区
├── 项目周报子工作区
├── 供应商询价子工作区
├── 设备故障分析子工作区
├── 培训资料生成子工作区
└── 客户需求整理子工作区
```

这些子工作区成熟后，可以进一步：

- 接入统一运行平台；
- 形成内部 Agent Service；
- 形成部门级应用；
- 形成可配置产品；
- 形成 Agent 驱动的 SaaS。

---

## 4. 产品目标

### 4.1 核心目标

1. 引导员工从真实工作开始，而不是从抽象 Agent 设计开始。
2. 允许员工在信息不完整的情况下启动。
3. 持续发现、记录、分类和关闭 unknown。
4. 将隐性经验逐步转化为组织资产。
5. 将稳定能力固化为 Skill。
6. 将多个相关 Skill 组合为受控 Harness。
7. 通过统一成熟度标准判断是否具备 Loop 条件。
8. 最终生成一个结构完整、可移交、可验证的子工作区。

### 4.2 成功定义

一个成功的子工作区应做到：

- 非创建者可以理解它；
- 非创建者可以运行它；
- 输入、输出和边界明确；
- 主要异常已被识别；
- 执行过程可以追踪；
- 结果可以被评价；
- 高风险动作有人工机制；
- 失败后可以停止、恢复或升级；
- 具备进入 Loop 的必要条件。

---

## 5. 非目标

本 PR 暂不重点解决：

- 最终 SaaS 的多租户、计费和商业化；
- 复杂的底层 Agent 框架选型；
- 模型训练和微调；
- 完整的企业权限系统实现；
- 具体数据库和接口技术方案；
- 一次性实现完全无人自治；
- 用 Agent 替代所有人工责任。

本 PR 的重点是：

> **定义一个可执行的工作孵化过程，以及母工作区应提供的功能和引导。**

---

## 6. 核心概念

## 6.1 母工作区

母工作区是用于创建、孵化和评估子工作区的引导环境。

它不直接完成某一个具体业务，而是帮助员工：

- 选择工作；
- 复现工作；
- 拆解任务；
- 发现 unknown；
- 创建 Skill；
- 验证 Skill；
- 组合 Harness；
- 判断 Loop readiness；
- 生成可交付的子工作区。

母工作区应像一个持续陪伴的“方法论教练 + 工作资产编辑器 + 成熟度评审器”。

---

## 6.2 子工作区

子工作区围绕一个明确业务目标建立，是最终可被业务人员或 Agent 使用的工作环境。

一个子工作区可以包含：

- 业务说明；
- 任务契约；
- Skills；
- 工作流程；
- 上下文；
- 案例；
- 评价标准；
- unknown 池；
- 人工节点；
- 异常处理规则；
- 运行记录；
- Loop readiness 说明。

---

## 6.3 Skill

Skill 是一个边界明确、可独立验证、可被复用的业务能力。

例如：

- 读取某类生产计划表；
- 校验物料编码；
- 提取会议决策；
- 计算标准工时；
- 生成项目风险清单；
- 对供应商报价进行结构化比较。

Skill 不等于一次成功的 Prompt。

一个成熟 Skill 至少应明确：

- 它解决什么问题；
- 接收什么输入；
- 产生什么输出；
- 什么情况下适用；
- 什么情况下不适用；
- 依赖哪些知识或工具；
- 如何判断成功；
- 遇到异常时如何处理。

---

## 6.4 Harness

Harness 是围绕一个业务目标，将多个 Skill、上下文、规则、状态、权限、评价和人工机制组合起来的受控工作系统。

可以将其理解为：

```text
Harness
=
Skills
+ 任务流程
+ 上下文
+ 状态
+ 规则
+ 权限
+ 评价
+ Human Gate
+ 异常处理
+ 运行记录
```

Harness 不是 Skill 的文件夹，而是一个完整的工作环境。

---

## 6.5 Loop-ready Harness

Loop-ready Harness 指尚不一定已经无人值守运行，但已经具备构建闭环所需的基础条件。

至少包括：

- 目标可定义；
- 当前状态可读取；
- 下一步行动可选择；
- 行动结果可验证；
- 进展可衡量；
- 有停止条件；
- 有预算和次数边界；
- 有异常升级机制；
- 有状态保存和恢复方式；
- 有人工接管点；
- 关键动作可审计。

---

## 6.6 Unknown

Unknown 是当前子工作区尚未明确，但可能影响正确性、稳定性、安全性或可复用性的未知项。

Unknown 不是失败，而是工作显化过程中的核心资产。

每发现一个 unknown，就意味着对真实工作多理解了一层。

常见 unknown 类型包括：

| 类型 | 示例 |
|---|---|
| 目标 Unknown | “做好报告”具体指什么？ |
| 输入 Unknown | 缺少某个字段时是否还能继续？ |
| 规则 Unknown | 同时出现两个优先级规则时哪个优先？ |
| 判断 Unknown | 什么情况下员工会否决系统建议？ |
| 异常 Unknown | 外部系统没有返回结果怎么办？ |
| 验收 Unknown | 谁判断结果正确，依据是什么？ |
| 权限 Unknown | Agent 是否可以直接发送、修改或删除？ |
| 工具 Unknown | 当前能力是否需要新接口或新工具？ |
| 状态 Unknown | 哪些信息需要跨轮次保存？ |
| 边界 Unknown | 哪些情况不属于该工作区处理范围？ |
| 风险 Unknown | 错误结果会造成什么影响？ |
| 责任 Unknown | 异常最终由谁负责？ |

---

## 7. 总体方法

母工作区通过五个阶段引导员工完成子工作区。

```text
阶段1：工作复现
阶段2：任务契约与 Skill 原型
阶段3：Skill 验证与边界完善
阶段4：Harness 组合与受控运行
阶段5：Loop readiness 认证
```

五个阶段不是纯线性流程。

在任何阶段，新的真实案例都可能重新打开 unknown，并使子工作区回到前一阶段补充。

核心循环为：

```text
执行真实任务
→ 发现 unknown
→ 补充事实或规则
→ 更新工作资产
→ 重新验证
→ 扩大适用范围
```

---

# 8. 阶段 1：工作复现

## 8.1 阶段目标

选择一个足够小、真实发生、能够在较短时间内完成的工作，让员工在母工作区中使用 Codex/OpenCode 复现。

此阶段不要求自动化，不要求形成 Skill，也不要求描述完整流程。

重点是：

> **让系统看到人实际上是怎么完成工作的。**

## 8.2 推荐任务特征

优先选择：

- 高频或重复发生；
- 输入和输出相对明确；
- 有真实历史案例；
- 员工本人熟悉；
- 结果容易检查；
- 错误可修正；
- 可以在较小范围试验。

不建议一开始选择：

- 整个岗位；
- 整个部门流程；
- 长周期战略工作；
- 高度依赖谈判和人际关系的工作；
- 错误不可逆的高风险任务；
- 没有任何验收方式的任务。

## 8.3 母工作区引导内容

母工作区应引导员工提供：

1. 这次具体要完成什么？
2. 为什么要做这件事？
3. 输入材料有哪些？
4. 最终交付物是什么？
5. 员工原本会怎么做？
6. 哪些步骤由 Codex/OpenCode 完成？
7. 哪些步骤员工进行了修改？
8. 为什么修改？
9. 哪些地方员工无法清楚解释？
10. 哪些信息如果缺失，任务就不能继续？

## 8.4 过程要求

系统应保存：

- 原始任务描述；
- 输入材料说明；
- 对话和操作过程；
- 人工补充的上下文；
- 人工修改前后的差异；
- 失败尝试；
- 最终结果；
- 员工对结果的评价；
- 首批 unknown。

## 8.5 阶段输出

- 一个完整的真实案例；
- 一份人工工作轨迹；
- 初始任务说明；
- 初始 unknown 池；
- 人工耗时基线；
- 初始成功标准。

## 8.6 晋级标准

满足以下条件即可进入阶段 2：

- 已完成至少一个真实任务；
- 有明确输入和输出；
- 能够说明任务的基本目标；
- 已记录人工修改；
- 已形成初始 unknown 列表；
- 最终结果已被业务人员确认可用。

不要求：

- 所有 unknown 已关闭；
- 流程已经稳定；
- Skill 已经完备。

---

# 9. 阶段 2：任务契约与 Skill 原型

## 9.1 阶段目标

将阶段 1 的单次任务，转化为一个初步可重复的任务契约，并将其中相对稳定的能力固化为 Skill 原型。

核心问题是：

> **如果明天再来一个类似任务，哪些内容可以直接复用？**

## 9.2 任务契约

母工作区应引导员工逐步明确：

- 业务目标；
- 输入；
- 输出；
- 前置条件；
- 主要步骤；
- 关键规则；
- 人工判断点；
- 验收标准；
- 已知异常；
- 禁止事项；
- 适用范围；
- 不适用范围；
- 责任人。

允许使用“不确定”“待补充”作为答案，但必须转化为 unknown。

## 9.3 Skill 原型划分

母工作区应帮助员工识别：

- 哪些动作是稳定且可复用的；
- 哪些动作可以独立测试；
- 哪些动作依赖大量上下文；
- 哪些动作本质上仍然是人工决策；
- 哪些动作不适合封装为 Skill。

Skill 原型应尽量满足：

- 单一目标；
- 输入输出明确；
- 边界可描述；
- 可以独立运行；
- 可以独立判断成功或失败；
- 对其他 Skill 的依赖有限。

## 9.4 Unknown 处理

此阶段重点发现：

- 规则不明确；
- 输入格式不稳定；
- 任务边界不清；
- 成功标准过于主观；
- 人工判断无法解释；
- 缺少工具；
- 缺少历史案例。

系统不应强迫员工立即解决所有 unknown，而应：

- 标记重要程度；
- 指定责任人；
- 建议补充证据；
- 关联具体案例；
- 指出它可能影响的 Skill 或步骤。

## 9.5 阶段输出

- 初版任务契约；
- 一个或多个 Skill 原型；
- Skill 的适用和不适用条件；
- 初版输入输出说明；
- 初版评价方法；
- 更新后的 unknown 池；
- 下一批需要验证的案例清单。

## 9.6 晋级标准

- 任务目标和输出可以被第三方理解；
- 至少形成一个 Skill 原型；
- Skill 可以在原始案例上重新运行；
- 原始结果可以被复现；
- 人工修改点已被记录；
- 关键 unknown 已分类；
- 已确定下一阶段需要测试的不同案例。

---

# 10. 阶段 3：Skill 验证与边界完善

## 10.1 阶段目标

通过更多真实案例验证 Skill，持续发现边界、异常和隐藏规则，使 Skill 从“能完成一次”进化为“可以稳定复用”。

核心问题是：

> **它在哪些情况下能够工作，又会在哪些情况下失败？**

## 10.2 案例扩展

母工作区应引导员工增加：

- 正常案例；
- 不同输入规模的案例；
- 边界案例；
- 缺失数据案例；
- 冲突数据案例；
- 历史失败案例；
- 权限不足案例；
- 外部系统异常案例；
- 结果容易产生歧义的案例。

## 10.3 每次验证应回答

1. Skill 是否完成任务？
2. 输出是否正确？
3. 输出是否稳定？
4. 是否出现人工修正？
5. 修正属于新规则还是偶然情况？
6. 是否发现新的 unknown？
7. 旧 unknown 是否被关闭？
8. Skill 的适用范围是否需要缩小或扩大？
9. 是否需要拆分 Skill？
10. 是否需要将确定性部分改为固定规则？

## 10.4 Unknown 关闭要求

Unknown 不能仅通过“有人说应该这样”关闭。

关闭 unknown 至少应具有一种证据：

- 正式业务规则；
- 多个真实案例；
- 业务负责人确认；
- 系统字段定义；
- 历史数据；
- 明确的风险决策；
- 可重复的验证结果。

关闭后还应记录：

- 最终结论；
- 依据；
- 影响范围；
- 是否需要加入 Skill；
- 是否需要加入评价；
- 是否需要加入异常处理。

## 10.5 Skill 成熟度判断

Skill 在进入 Harness 前，应基本满足：

- 多个案例验证；
- 输入输出稳定；
- 适用范围清晰；
- 常见异常已识别；
- 关键规则已显化；
- 结果可以评价；
- 失败可以被识别；
- 非创建者可以运行；
- 版本变化可以说明；
- 关键 unknown 已关闭或有明确处理方式。

## 10.6 阶段输出

- 成熟 Skill；
- Skill 案例集；
- Skill 评价标准；
- 错误类型清单；
- 异常处理建议；
- 适用边界；
- 剩余 unknown；
- Skill 之间可能的组合关系。

## 10.7 晋级标准

- 至少使用多个不同真实案例验证；
- 主要正常场景可稳定完成；
- 常见失败可以被识别；
- 结果有明确的检查方式；
- 关键规则不再只存在于个人脑中；
- 非创建者能够理解并运行；
- Skill 的边界已经足够清晰，可以进入组合。

---

# 11. 阶段 4：Harness 组合与受控运行

## 11.1 阶段目标

将围绕同一业务目标的多个 Skill 组合为一个子工作区，并加入公共上下文、流程、状态、权限、评价和人工机制。

核心问题是：

> **这些能力如何共同完成一个完整工作，而不是分别完成几个动作？**

## 11.2 组合原则

只有满足以下条件的 Skill 才应被组合：

- 服务于同一个业务目标；
- 有明确前后依赖；
- 共享相近上下文；
- 使用相近的数据和工具；
- 可以用共同业务结果进行评价；
- 权限边界可以统一治理。

## 11.3 Harness 需要明确的内容

### 业务目标

- Harness 最终要完成什么；
- 什么状态代表工作完成；
- 哪些结果只能作为建议；
- 哪些结果可以直接执行。

### 流程

- 起点；
- 主要步骤；
- 判断节点；
- 并行或串行关系；
- 人工审批点；
- 异常分支；
- 结束状态。

### 上下文

- 固定业务规则；
- 当前任务输入；
- 组织知识；
- 用户偏好；
- 历史状态；
- 临时工作文件；
- 哪些信息可以长期保留；
- 哪些信息只能本次使用。

### 权限

- 只读动作；
- 可修改动作；
- 可对外发送动作；
- 不可逆动作；
- 必须审批动作；
- 严禁动作。

### 评价

- 每个 Skill 的局部评价；
- Harness 的最终业务评价；
- 人工抽检方法；
- 失败判定；
- 可接受偏差。

### 异常

- 输入缺失；
- 规则冲突；
- Skill 失败；
- 外部工具失败；
- 权限不足；
- 结果不确定；
- 业务目标不可达；
- 人工拒绝。

## 11.4 自治级别

Harness 应支持逐级运行，而不是直接无人值守：

| 等级 | 运行方式 |
|---|---|
| H0 观察 | 人执行，工作区只记录和分析 |
| H1 建议 | Harness 生成建议，人完成操作 |
| H2 沙箱 | Harness 执行，但不写入正式系统 |
| H3 审批执行 | Harness 准备动作，人确认后执行 |
| H4 异常接管 | Harness 正常运行，人处理异常 |
| H5 有界自治 | Harness 在明确边界内自主执行 |

母工作区应根据风险和成熟度推荐当前自治等级。

## 11.5 阶段输出

- 一个完整子工作区；
- Harness 业务说明；
- Skill 组合关系；
- 工作流程；
- 上下文清单；
- 状态说明；
- 权限和人工节点；
- 异常处理机制；
- 评价方式；
- 运行案例；
- 当前自治等级；
- 剩余 unknown。

## 11.6 晋级标准

- 可以完成端到端业务流程；
- 每一步输入输出可以追踪；
- Skill 组合后通过端到端验证；
- 关键人工节点已经明确；
- 高风险动作已有控制方式；
- 常见异常已有处理路径；
- Harness 可以由非创建者运行；
- 运行结果可以被业务指标评价；
- 具备继续评估 Loop 条件的基础。

---

# 12. 阶段 5：Loop Readiness 认证

## 12.1 阶段目标

判断 Harness 是否具备构建持续闭环的条件，并补齐缺失项。

本阶段的目标不是强制开启无人 Loop，而是交付一个：

> **具备 Loop 条件、能够安全进入闭环设计的完备 Harness。**

## 12.2 Loop Readiness 十项条件

### 1. Goal：目标

- 目标是否明确；
- 目标是否可以被系统读取；
- 目标是否可以被拆分；
- 是否存在互相冲突的目标。

### 2. State：状态

- 当前状态是否可以保存；
- 是否知道已完成什么；
- 是否知道下一轮从哪里继续；
- 中断后是否可以恢复。

### 3. Action：行动

- 可执行的动作是否来自已验证 Skill；
- 每个动作的输入输出是否明确；
- 是否知道动作的风险和副作用。

### 4. Evaluator：评价

- 是否可以判断每一步是否成功；
- 是否可以判断距离目标是否更近；
- 是否有最终验收方法；
- 是否避免完全依赖 Agent 自我评价。

### 5. Stop：停止

- 什么情况下目标完成；
- 什么情况下任务失败；
- 什么情况下必须暂停；
- 什么情况下必须转人工。

### 6. Budget：预算

- 最大运行次数；
- 最大时间；
- 最大成本；
- 最大工具调用量；
- 是否存在单次动作限制。

### 7. Retry：重试

- 哪些动作可以重试；
- 最多重试几次；
- 是否会重复产生副作用；
- 是否有去重机制。

### 8. Escalation：升级

- 哪些异常交给谁；
- 需要提供哪些上下文；
- 人工处理后如何回到流程；
- 未知异常是否自动进入 unknown 池。

### 9. Recovery：恢复

- 中断后如何继续；
- 错误写入如何回滚；
- 如何恢复到安全状态；
- 是否可以重放执行过程。

### 10. Observability：可观测性

- 能否看到每一步发生了什么；
- 能否看到使用了哪些输入、规则和 Skill；
- 能否解释为什么做出某个动作；
- 能否定位失败原因；
- 能否比较不同版本表现。

## 12.3 Loop Readiness 结果

认证结果可以分为：

| 结果 | 含义 |
|---|---|
| 未准备 | 关键目标、状态或评价缺失 |
| 条件准备 | 可进行人工驱动的闭环试验 |
| 受控准备 | 可在审批或沙箱条件下运行 Loop |
| 有界准备 | 可在限定范围内自主运行 |
| 生产候选 | 已具备进入统一 Agent 运行平台的条件 |

## 12.4 阶段输出

- Loop-ready Harness；
- Loop readiness 检查结果；
- 目标定义；
- 状态模型；
- 评价器说明；
- 停止条件；
- 预算边界；
- 重试和去重规则；
- 人工升级机制；
- 恢复机制；
- 可观测性要求；
- 未关闭 unknown；
- 不允许自动化的边界；
- 推荐的下一步产品化路径。

## 12.5 最终交付标准

子工作区达到以下条件，才可被标记为“完备”：

- 业务目标清晰；
- 任务契约完整；
- 核心 Skill 已验证；
- Harness 已通过端到端案例；
- Unknown 已被管理，而不是被忽略；
- 未关闭 unknown 有明确影响和责任人；
- 输入、输出、状态和边界明确；
- 结果可以评价；
- 关键异常可识别；
- 高风险动作有人工机制；
- 有停止、恢复和升级机制；
- 非创建者可理解、运行和维护；
- 具备 Loop readiness 结论。

# 13. Unknown 管理机制

## 13.1 Unknown 生命周期

每个 unknown 应经过：

```text
发现
→ 描述
→ 分类
→ 判断影响
→ 分配责任人
→ 收集证据
→ 得出结论
→ 固化到工作资产
→ 验证
→ 关闭
```

已经关闭的 unknown 可以因为新案例重新打开。

## 13.2 Unknown 状态

- 新发现；
- 待澄清；
- 待证据；
- 待业务确认；
- 待验证；
- 已解决；
- 已接受风险；
- 已转人工；
- 不在范围；
- 重新打开。

## 13.3 Unknown 优先级

建议按以下因素判断：

- 对结果正确性的影响；
- 对安全的影响；
- 对自动化边界的影响；
- 出现频率；
- 是否阻塞下一阶段；
- 是否可能造成不可逆后果。

## 13.4 Unknown 与资产关联

每个 unknown 应能关联：

- 具体案例；
- 任务契约；
- Skill；
- Harness 步骤；
- 评价项；
- 权限；
- 异常类型；
- Loop readiness 条件。

## 13.5 Unknown 不必全部关闭

部分 unknown 可以长期存在，但必须明确：

- 它是什么；
- 何时会出现；
- 出现后怎么办；
- 是否转人工；
- 是否接受风险；
- 是否限制自动化范围。

一个成熟系统不是“没有 unknown”，而是：

> **知道哪些是 unknown，并且知道如何处理。**

---

# 14. 母工作区的核心功能

## 14.1 新建子工作区

引导用户填写：

- 工作名称；
- 工作目的；
- 当前负责人；
- 真实案例；
- 预期收益；
- 风险等级；
- 当前所处阶段。

## 14.2 阶段导航

系统应始终展示：

- 当前阶段；
- 已完成内容；
- 缺失内容；
- 关键 unknown；
- 推荐下一步；
- 晋级条件；
- 不建议提前做的事项。

## 14.3 工作复现引导

帮助员工：

- 选择小任务；
- 上传或说明输入；
- 使用 Codex/OpenCode 完成；
- 标记人工修改；
- 比较前后差异；
- 提取首批 unknown。

## 14.4 任务契约生成

根据真实案例生成任务契约草稿，并通过提问逐步完善。

系统不得假装所有信息都已明确。

无法确认的内容必须进入 unknown。

## 14.5 Skill 候选识别

从工作轨迹中建议：

- 哪些步骤适合成为 Skill；
- 哪些步骤应保留为规则；
- 哪些步骤应保留人工判断；
- 哪些步骤依赖新工具；
- 哪些 Skill 可能重复。

最终是否创建 Skill 由员工或负责人确认。

## 14.6 案例与评价管理

支持维护：

- 正常案例；
- 边界案例；
- 失败案例；
- 黄金案例；
- 人工预期结果；
- 评价标准；
- 回归验证结果。

## 14.7 Unknown 池

支持：

- 自动提取 unknown；
- 人工新增；
- 分类；
- 关联资产；
- 指派负责人；
- 记录证据；
- 关闭和重开；
- 显示对成熟度的影响。

## 14.8 Harness 组合引导

帮助用户定义：

- 总体目标；
- Skill 顺序；
- 输入输出关系；
- 公共上下文；
- 状态；
- 人工节点；
- 权限；
- 异常分支；
- 最终评价。

## 14.9 成熟度仪表板

展示：

- 当前阶段；
- Skill 成熟度；
- 案例覆盖度；
- Unknown 数量和风险；
- Harness 完整度；
- 当前自治等级；
- Loop readiness；
- 阻塞项；
- 推荐下一步。

成熟度不应只由文档是否填写决定，而应综合：

- 真实案例；
- 运行结果；
- 人工反馈；
- 错误类型；
- 未关闭 unknown；
- 业务负责人确认。

## 14.10 子工作区生成

最终生成的子工作区应包含：

```text
子工作区
├── README：业务说明与使用入口
├── TASK-CONTRACT：任务契约
├── SKILLS：核心能力
├── CASES：真实案例与黄金案例
├── EVALS：评价方法与结果
├── WORKFLOW：Harness 流程
├── CONTEXT：上下文说明
├── STATE：状态说明
├── POLICIES：权限与行为边界
├── HUMAN-GATES：人工审批与接管
├── EXCEPTIONS：异常类型与处理
├── UNKNOWNS：Unknown 池
├── LOOP-READINESS：Loop 条件检查
└── CHANGELOG：演化记录
```

该结构表示逻辑内容，不限定具体技术目录实现。

# 15. 关键交互原则

## 15.1 不要求一次填完

系统必须允许用户从不完整状态开始。

推荐使用：

- “当前不知道”；
- “需要案例验证”；
- “需要负责人确认”；
- “暂时保留人工”；
- “不在当前范围”。

这些选项应自动创建 unknown，而不是阻塞用户。

## 15.2 以真实案例代替抽象访谈

当用户无法回答规则时，系统优先询问：

- 能否提供一个实际案例？
- 当时你做了什么？
- 为什么这样修改？
- 有没有相反案例？
- 如果不修改会怎样？

## 15.3 显示进步，而不是只显示缺陷

系统应向用户展示：

- 本轮关闭了哪些 unknown；
- 新显化了哪些规则；
- 哪些人工动作已经可以交给 Skill；
- 哪些风险已被控制；
- 子工作区成熟度如何提升。

## 15.4 不鼓励虚假完备

系统不应为了完成表单而自动编造规则。

任何没有依据的结论，应标记为：

- 假设；
- 待验证；
- 低置信度；
- 暂定规则。

## 15.5 不以“全自动”为唯一目标

母工作区应允许最终结果是：

- 人工主导；
- Agent 建议；
- 人工审批；
- Agent 正常运行、人工处理异常；
- 有界自治。

部分工作长期保留 Human in the Loop 仍然可以被视为成熟。

# 16. 角色与协作

## 16.1 工作实践者

最熟悉真实工作的人。

负责：

- 提供案例；
- 复现任务；
- 解释人工修改；
- 发现 unknown；
- 验证输出是否可用。

## 16.2 业务 Owner

负责：

- 定义业务目标；
- 确认规则；
- 决定风险边界；
- 确认验收标准；
- 批准自治等级。

## 16.3 Skill / Harness 维护者

负责：

- 维护 Skill；
- 维护案例和评价；
- 处理版本变化；
- 分析失败；
- 维护子工作区。

## 16.4 评审者

负责：

- 检查是否满足晋级条件；
- 检查 unknown 是否被掩盖；
- 检查结果是否可被独立验证；
- 检查高风险动作是否有人工机制。

## 16.5 平台管理员

负责：

- 提供母工作区；
- 维护通用模板；
- 管理共享 Skill；
- 管理工作区目录；
- 监控复用、质量和价值。

# 17. 关键用户故事

## 17.1 从真实任务开始

作为一名员工，我可以选择刚完成的一项真实工作，将输入、过程、人工修改和结果交给母工作区，使系统帮助我建立初始子工作区。

## 17.2 不知道也可以继续

作为一名员工，当我无法解释某条业务规则时，我可以选择“unknown”，系统记录并引导我后续通过案例或负责人确认，而不是阻塞整个流程。

## 17.3 自动提取人工判断

作为一名员工，系统可以从我对 Codex 输出的修改中提示：“这里可能存在一条隐性规则”，并让我确认是否加入 unknown 或任务契约。

## 17.4 形成 Skill

作为一名员工，系统可以从已复现的任务中识别可复用步骤，帮助我将其定义为 Skill 候选，并说明它的输入、输出和边界。

## 17.5 用案例验证 Skill

作为一名维护者，我可以为 Skill 增加不同真实案例，查看在哪些情况下成功、失败或需要人工干预。

## 17.6 组合 Harness

作为一名业务 Owner，我可以将多个成熟 Skill 组合成一个完整业务流程，明确人工审批、异常处理和最终验收。

## 17.7 查看成熟度

作为一名评审者，我可以查看当前子工作区所处阶段、未关闭 unknown、案例覆盖度、风险和阻塞项。

## 17.8 判断是否具备 Loop 条件

作为一名业务 Owner，我可以查看 Harness 是否具备目标、状态、评价、停止、预算、升级和恢复机制，并决定是否进入 Loop 试验。

## 17.9 交给其他人使用

作为一名同事，我可以进入别人创建的子工作区，理解它完成什么、如何使用、有哪些边界，以及遇到异常时找谁。

# 18. 验收标准

## 18.1 母工作区验收

母工作区至少应支持：

- 创建子工作区；
- 选择并记录当前阶段；
- 引导完成真实任务复现；
- 记录人工修改；
- 建立任务契约；
- 创建和管理 Skill 候选；
- 管理案例和评价；
- 创建和管理 unknown；
- 组合 Harness；
- 定义人工节点和异常；
- 进行 Loop readiness 检查；
- 生成完整子工作区；
- 展示成熟度和下一步建议。

## 18.2 方法论验收

使用母工作区后，应能够观察到：

- 员工不需要一次性描述完整流程；
- Unknown 数量会先增加，再逐步被关闭或管理；
- 人工修改可以转化为规则、案例或判断节点；
- Skill 不再只是 Prompt；
- Harness 不再只是 Skill 集合；
- 最终子工作区可以被其他人理解和使用；
- Loop readiness 有清晰判断依据；
- 未达到条件时，系统会明确阻止过早自治。

## 18.3 子工作区验收

至少选取一个真实业务任务进行试点。

试点子工作区应满足：

- 至少一个完整真实案例；
- 至少一个任务契约；
- 至少一个经过多案例验证的 Skill；
- 至少一个端到端 Harness；
- 完整 unknown 池；
- 明确的人工节点；
- 明确的异常处理；
- 完成一次独立人员复现；
- 形成 Loop readiness 结果。

# 19. 建议的试点方式

## 19.1 选择试点任务

优先从以下任务中选择：

- 周报生成；
- Excel 数据整理；
- 项目风险汇总；
- 会议纪要和行动项提取；
- 产能基础分析；
- 标准文档生成；
- 重复性信息核对。

## 19.2 试点人数

建议：

- 1 名工作实践者；
- 1 名业务 Owner；
- 1 名工作区维护者；
- 1 名独立使用者。

## 19.3 试点目标

不是立即实现无人自动化，而是验证：

1. 母工作区是否能引导员工完成五阶段；
2. Unknown 是否真正被发现和显化；
3. Skill 是否能被非创建者使用；
4. Harness 是否能完成端到端任务；
5. Loop readiness 是否能够被客观判断。

# 20. 风险与边界

## 20.1 用户为了晋级而填写虚假信息

风险：

- 为了“完成阶段”，用户可能随意补全规则；
- 文档看起来完整，但没有真实依据。

应对原则：

- 区分事实、假设和 unknown；
- 关键结论要求关联案例或负责人确认；
- 晋级依据不能只看填写完整度。

## 20.2 Skill 数量膨胀

风险：

- 大量重复 Skill；
- 粒度过大或过小；
- 无人维护。

应对原则：

- 优先复用已有 Skill；
- 创建前提示相似 Skill；
- 要求说明适用边界；
- 用复用率和成功率，而不是数量评价。

## 20.3 过早组合 Harness

风险：

- 不成熟 Skill 被组合；
- 局部错误在流程中放大。

应对原则：

- Skill 必须经过阶段 3；
- Harness 必须有端到端案例；
- 局部评价和最终评价都必须存在。

## 20.4 过早开启 Loop

风险：

- 无限重试；
- 重复写入；
- 错误自我强化；
- 成本失控；
- 无人负责。

应对原则：

- 未通过 Loop readiness 不进入 Loop；
- 必须有停止、预算、升级、恢复和可观测性；
- 高风险动作保留人工机制。

## 20.5 Unknown 被当作缺陷隐藏

风险：

- 员工担心 unknown 影响成熟度；
- 关键问题没有被记录。

应对原则：

- 奖励发现高价值 unknown；
- Unknown 的存在不等于不成熟；
- 未知但可控优于假装已知。

## 20.6 工作区只对创建者有效

风险：

- 大量隐性上下文仍然留在创建者脑中。

应对原则：

- 每阶段安排非创建者复现；
- 最终交付必须通过独立使用者验证；
- 无法移交则不能标记完备。

# 21. 未来产品化路径

当多个员工使用母工作区创建出成熟子工作区后，可以逐步形成：

## 21.1 共享 Skill 生态

- 通用 Skill 目录；
- 部门 Skill 目录；
- 复用关系；
- 版本和责任人；
- 案例和成熟度；
- 推荐与搜索。

## 21.2 Harness 模板

将相似子工作区提炼为模板，例如：

- 报告生成 Harness；
- 数据核对 Harness；
- 审批准备 Harness；
- 计划分析 Harness；
- 异常处理 Harness。

## 21.3 Agent 运行平台

对成熟 Harness 提供：

- 统一触发；
- 统一身份；
- 统一权限；
- 统一任务状态；
- 统一日志；
- 统一人工接管；
- 统一成本与效果统计。

## 21.4 内部产品

将多个成熟 Harness 组合成面向特定角色的内部应用。

## 21.5 Agent SaaS

当某类 Harness：

- 有多个稳定用户；
- 需求高度相似；
- 配置可以替代大量定制；
- 价值可以量化；
- 运行和维护成本可控；

再进一步产品化为 Agent 驱动的 SaaS。

# 22. 核心原则

本 PR 的方法论可以浓缩为以下原则：

1. **从真实工作开始，不从抽象 Agent 开始。**
2. **从小任务开始，不直接自动化整个岗位。**
3. **允许不知道，但必须记录 unknown。**
4. **每次人工修改，都是潜在知识。**
5. **成功一次不是 Skill，多案例稳定才是 Skill。**
6. **Skill 是能力，Harness 是工作系统。**
7. **Harness 成熟不等于立即无人自治。**
8. **没有评价，就不能扩大权限。**
9. **没有停止、升级和恢复，就不能进入 Loop。**
10. **Unknown 不需要全部消失，但必须可见、可控、可负责。**
11. **最终交付必须能被非创建者使用。**
12. **产品化建立在稳定复用之上，而不是建立在演示成功之上。**

# 23. 一句话定义

> **该母工作区通过五阶段引导员工从一个真实小任务出发，持续发现和关闭 unknown，将隐性工作逐步固化为 Skill、组合为 Harness，并最终交付一个可验证、可移交、可治理、具备 Loop 条件的完备子工作区。**
